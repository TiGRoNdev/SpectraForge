Коротко по фактам:
- Изображение: `triangle_splatting_frame.ppm`, 960×540, mean≈96.87 → контент есть, чёрного экрана нет.
- Логи: кадры ~1.2–1.6s при 960×540 → узкое место в compute-пайплайне.

Основные причины низкой производительности:
- О(N×M): в `TriangleSplatting.comp` на пиксель итерируются все треугольники; тайловый список используется лишь при `enableTileBinning=1`. Сейчас fallback активен (по умолчанию 0).
- Видимость/сортировка: фрустум- и тайл-списки не влияют на порядок и объём работ рендер-прохода; сортировка делается “по всем”, а не “по видимым/по тайлу”.
- Нет opaque/Z-path: даже непрозрачные треугольники проходят дорогой блендинг.
- Инициализация счётчиков/буферов видимости кадр-за-кадром не обнулена (риски “грязных” данных и неоптимальной работы).
- Проекция/SDF: визуально кадр есть, но нет backface culling в экранном пространстве — лишняя работа.

Что сделать (быстрые, безопасные правки):
- Включить тайловый биннинг:
  - Установить `pushConstants.enableTileBinning = 1` после успешного Dispatch `TileCulling` и барьера (shader read).
  - Убедиться, что цикл в `TriangleSplatting.comp` всегда берёт `tileTriangles[tileIndex]` при enable=1; при переполнении тайла — fallback на `sortedIndices` только для избытка.
  - Добавить барьер после тайл-прохода: write(tileBuffer) → read(tileBuffer) в основном проходе.
- Сортировка по глубине на видимом подмножестве:
  - Считать depth-keys и сортировать не весь массив, а `visibleIndices`/тайловые индексы, локально per-tile (front-to-back). Это резко уменьшит блендинг и повысит ранний выход по альфе.
- Сброс счётчиков каждый кадр:
  - `vkCmdFillBuffer(atomicCounterBuffer_, 0)`, `vkCmdFillBuffer(visibleIndicesBuffer_, 0xFFFFFFFF)` и барьер до culling/indirect.
- Opaque fast path:
  - Для треугольников с `opacity >= 0.98` включить непрозрачный путь: писать в отдельный depth image (или имитировать Z-тест через min-depth на пиксель с subgroup reduce), без альфа-бленда. Это сильно снижает смешивания.
- Backface culling в экранном пространстве:
  - Быстрый знак площади (cw/ccw) по проекциям v0,v1,v2; отбрасывать невидимые.
- Динамический бюджет:
  - Регулировать `triangleStep`/разрешение по `frame time` (простая DRR): при >100ms — увеличить step или снизить scale, при <50ms — вернуть.

Среднесрочные улучшения (даёт ещё 5–20×):
- Two-pass visibility→shading:
  - Pass 1: маркировать “касаемые” пиксели/тайлы и индексы треугольников (ID/raster), Pass 2: shade только по спискам.
- Перепаковка данных:
  - Хранить предварительно просчитанные экранные координаты треугольника и параметры окна (AABB, нормали рёбер) в компактном буфере — меньше проекций/чтений внутри пиксельного цикла.
- BVH/иерархический culling для треугольников до биннинга (CPU или GPU).

Проверки/валидация:
- Перед основным проходом: барьеры для `outputImage` (GENERAL), тайл-буфера (shader write→read), видимых индексов, atomic counter (fill→shader write).
- В логах: перестанет печататься “bound only 3 descriptors”, появится включение тайлов и заметно упадёт время кадра.
- Метрики HOWTO: целимся к ≤100–150ms @960×540 после включения тайлов и локальной сортировки.
