## 🧮 Математическая модель проблемы

### 1. Требование непрерывного покрытия

**Теорема покрытия** (из Perplexity):
> Для визуально непрерывной поверхности расстояние между соседними гауссианами должно быть:
> 
> **d ≤ 2.5σ**
> 
> где d — расстояние между центрами, σ — среднеквадратичное отклонение.

**Доказательство**:
Гауссиан: `G(x) = exp(-x² / (2σ²))`

На расстоянии `2.5σ` от центра: `G(2.5σ) = exp(-6.25/2) ≈ 0.04` (4% от пика)

Для непрерывности нужно, чтобы **соседние гауссианы перекрывались** на уровне > 10%.

При `d = 2.5σ`:
```
Гауссиан A:     ╱‾‾╲       
Гауссиан B:         ╱‾‾╲
Перекрытие:      ▓▓▓      ← 10-15% область
```

---

### 2. Наш текущий алгоритм

**Шаг 1: Subdivision треугольника**

Для треугольника с вершинами `(v0, v1, v2)` и площадью `A`:

```
subdivisionLevel = {
    9,  если A > 0.08
    6,  если 0.05 < A ≤ 0.08
    4,  если 0.03 < A ≤ 0.05
    3,  если 0.015 < A ≤ 0.03
    2,  если A ≤ 0.015
}
```

**Размер гауссиана**:
```cpp
sigma = sqrt(A / subdivisionLevel) * 0.7
sigma = clamp(sigma, 0.025, 0.2)
```

**Проблема**: Это дает **фиксированное количество** точек на треугольник, **независимо от его проекции на экран**!

---

### 3. Проекция на экран

Треугольник в 3D: площадь `A_world`  
После проекции: площадь `A_screen = A_world * |det(ViewProj)| / z²`

**Пример**:
- Треугольник в world space: `A = 0.1 м²`
- Расстояние от камеры: `z = 5 м`
- FOV = 60°, экран 1920×1080

Проекция:
```
scale = (1920 / tan(30°)) / z = 1920 / (5 * 0.577) ≈ 665 px/m

A_screen ≈ 0.1 * 665² ≈ 44,000 пикселей
```

С 9 гауссианами:
```
Плотность: 9 / 44,000 = 0.0002 гауссиана/пиксель
Для покрытия нужно: 1 / (π * (2.5σ)²) гауссианов/пиксель

При σ_screen = 1.6 px:
1 / (π * 4²) ≈ 0.02 гауссиана/пиксель

Дефицит: 0.02 / 0.0002 = 100x !!!
```

**Вывод**: Нужно **в 100 раз больше** гауссианов на этот треугольник!

```
┌─────────────────────────────────────────────────────────────────┐

└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ 3. Upload to GPU                                                │
│    SSBO: GaussianSplat { vec4 posAndScale, vec4 colorWeight }  │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ 4. Compute Shader (GPU)                                         │
│    For each pixel (x,y):                                        │
│      For each Gaussian g:                                       │
│        - Project: screenPos = ViewProj * g.position             │
│        - Distance: d = length(screenPos - pixel)                │
│        - Weight: w = exp(-d² / (2σ²))                           │
│        - Accumulate: color += g.color * w                       │
│    Output: Sparse dots on contours                              │
└─────────────────────────────────────────────────────────────────┘
```

---

### Проблемы на каждом шаге:

**Проблема 1 (Subdivision)**: Не учитывает **screen-space density**
- Треугольник близко к камере: большая проекция → нужно МНОГО гауссианов
- Треугольник далеко: маленькая проекция → достаточно МАЛО гауссианов
- Наш алгоритм: **одинаковое** subdivision для обоих!

**Проблема 2 (Sampling strategy)**: Только **барицентры и рёбра**
- Это даёт точки на **контуре** треугольника
- Для заполнения нужны точки **внутри** (регулярная сетка)

**Проблема 3 (No optimization)**: Фиксированные позиции
- 3D Gaussian Splatting (NeRF) **оптимизирует** позиции и размеры
- Наш подход: **одноразовое** размещение, без адаптации

## 📊 Сравнение подходов

### Подход 1: Наш текущий (Triangle Barycenter Splatting)

```
Input: Triangle mesh
↓
Sample barycenters (2-9 per triangle)
↓
Project to screen
↓
Render as 2D Gaussians
```

**Математика**:
- Количество гауссианов: `N_tri × subdivision_level`
- Покрытие экрана: `N_gaussians × π × (2.5σ)²`
- Для Sponza: `66,000 × 3.4 ≈ 225,000` гауссианов
- Покрытие: `225,000 × π × 4² ≈ 11,000,000 px²` из `2,073,600 px` → **532% overdraw**

**НО!** Гауссианы **не равномерны** → большие пробелы на поверхностях, скопления на рёбрах.

---

### Подход 2: 3D Gaussian Splatting (NeRF-based)

```
Input: Multi-view images OR mesh
↓
Initialize: Dense point cloud (millions of points)
↓
Optimize: Positions, scales, colors, opacities (gradient descent)
↓
Render: Anisotropic 3D → 2D projection
```

**Математика**:
- Количество гауссианов: `5-10 миллионов` (оптимизированы!)
- Каждый гауссиан: анизотропная ковариация `3×3`
- Оптимизация: `∇L = ∑ (rendered - gt)²` по всем позициям/размерам
- Результат: **адаптивная плотность** (плотнее на деталях, разреженнее на плоскостях)

**Ссылки** (из Perplexity):
- [MeshSplats](https://arxiv.org/html/2502.07754v1): Hybrid mesh+splat rendering
- [SuGaR](https://github.com/Anttwo/SuGaR): Surface-aligned Gaussian optimization
- [GaMeS](https://github.com/waczjoan/gaussian-mesh-splatting): Mesh-based Gaussian adaptation

---

### Подход 3: Traditional Rasterization

```
Input: Triangle mesh
↓
For each triangle:
  - Project vertices to screen
  - Rasterize: fill all pixels inside triangle
  - Interpolate: colors, normals, UVs
↓
Z-buffer: handle occlusion
↓
Output: Filled surfaces (exact coverage)
```

**Математика**:
- Покрытие: **100%** пикселей внутри треугольника (по определению)
- Производительность: `~100 миллионов` треугольников/сек на современных GPU
- Сложность: `O(N_triangles × pixels_per_triangle)`

## 🔬 Математические формулы

### Требуемая плотность гауссианов

Для сплошного покрытия экрана:

```
N_required = A_screen / (π × (2.5σ_screen)²)
```

Где:
- `A_screen` — площадь проекции на экран (пиксели²)
- `σ_screen` — стандартное отклонение в пикселях

**Пример для Sponza**:
- Средняя площадь проекции треугольника: `20 px²`
- `σ_screen = 2 px`
- `N_required = 20 / (π × 5²) ≈ 0.25` гауссиана на треугольник

**НО!** Это для **маленьких** треугольников. Для больших (300×200 px):
- `A_screen = 60,000 px²`
- `N_required = 60,000 / (π × 5²) ≈ 764` гауссиана на треугольник!

**Наш алгоритм даёт**: 9 гауссианов → **дефицит 85×**

### Адаптивное subdivision (правильный подход)

```cpp
// 1. Проецируем вершины треугольника
vec4 clip0 = ViewProj * vec4(v0, 1.0);
vec4 clip1 = ViewProj * vec4(v1, 1.0);
vec4 clip2 = ViewProj * vec4(v2, 1.0);

vec2 screen0 = (clip0.xy / clip0.w) * 0.5 + 0.5;
vec2 screen1 = (clip1.xy / clip1.w) * 0.5 + 0.5;
vec2 screen2 = (clip2.xy / clip2.w) * 0.5 + 0.5;

// 2. Площадь в screen space
float A_screen = abs(cross(screen1 - screen0, screen2 - screen0)) * 0.5 * width * height;

// 3. Адаптивное subdivision
float target_density = 0.02; // гауссианов/пиксель
int subdivision_level = ceil(A_screen * target_density);

// 4. Регулярная сетка внутри треугольника
for (int i = 0; i < subdivision_level; ++i) {
    // Barycentric координаты случайно или регулярно
    vec3 bary = generate_barycentric(i, subdivision_level);
    vec3 world_pos = bary.x * v0 + bary.y * v1 + bary.z * v2;
    add_gaussian(world_pos, sigma);
}
```

## 📈 Выводы из математического анализа

### 1. **Fundamental Problem: Sparse Sampling**

Барицентрическое семплирование треугольников **не даёт** достаточной плотности для сплошных поверхностей.

**Решение**:
- Dense point cloud sampling (миллионы точек)
- ИЛИ традиционная растеризация

---

### 2. **Screen-Space Density Mismatch**

Subdivision в **world space** не учитывает **screen-space** проекцию.

**Решение**:
- Адаптивное subdivision по проекции
- LOD (Level of Detail) по расстоянию

---

### 3. **No Optimization**

Фиксированные позиции гауссианов **не адаптируются** к геометрии.

**Решение**:
- Gradient-based optimization (как в NeRF)
- ИЛИ use rasterization

---

### 4. **Performance vs Quality Tradeoff**

| Метод | Качество | Производительность | Подходит для Sponza? |
|-------|----------|-------------------|---------------------|
| Наш GS (225K) | ❌ Контуры | ✅ 60 FPS | ❌ Нет |
| Плотный GS (10M) | ✅ Сплошные | ⚠️ 30 FPS | ⚠️ С оптимизацией |
| Rasterization | ✅ Идеально | ✅ 200+ FPS | ✅ Да |
